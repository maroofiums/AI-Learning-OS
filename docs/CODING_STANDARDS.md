# Coding Standards — AI Learning OS

This codifies the rules from the project's development instructions into concrete,
checkable practices. Treat this as a checklist during code review (see "Code Reviews"
in the dev instructions).

---

## 1. Python Style

- Follow PEP8. Enforce with `ruff` (lint) and `black` (format) — add both to
  `requirements.txt` and run in a pre-commit hook.
- Type hints are mandatory on every function signature, including return types.
  `mypy` runs in CI/pre-commit; treat type errors as build failures, not warnings.
- Naming: `snake_case` for functions/variables, `PascalCase` for classes,
  `UPPER_SNAKE_CASE` for constants. No abbreviations that aren't obvious
  (`usr` → `user`, `proj` → `project`).
- Docstrings (Google style) required on every public class and function in
  `service.py`, `repository.py`, and `router.py` files. Private helpers (`_foo`) are
  exempt unless the logic is non-obvious.

## 2. Layered Architecture Rules (enforced, not optional)

- **Routers never touch the database.** A `router.py` function may only call a
  `service.py` function and return its result (mapped through a Pydantic schema).
- **Services never construct SQL/ORM queries directly.** That belongs in
  `repository.py`. Services orchestrate business logic and call repository methods.
- **Repositories never contain business logic.** They are thin — fetch, insert,
  update, delete, with query construction only.
- **AI modules (`backend/ai/*`) never import from `backend/modules/*` directly.**
  Domain modules call AI services through a defined interface (e.g.
  `ai.rag.service.answer_question(query, user_id)`), not the other way around. This
  keeps AI logic swappable per ARCHITECTURE.md's stated goal.

If you find yourself writing a database query inside a router, or business logic
inside a repository, stop and move it to the correct layer before continuing.

## 3. Module Structure (every domain module follows this exact shape)

```
modules/<name>/
    __init__.py
    router.py      # FastAPI routes, request/response wiring only
    service.py      # business logic, orchestration
    repository.py    # database access for this module's tables
    schemas.py        # Pydantic request/response models
    models.py          # SQLAlchemy ORM models
    tests/
        __init__.py
        test_service.py
        test_repository.py
        test_router.py
```

Do not add files outside this shape without a documented reason in the module's own
README (create one if a module grows complex enough to need it).

## 4. Error Handling

- Raise custom exceptions defined in `core/exceptions.py` (e.g. `NotFoundError`,
  `ValidationError`, `UnauthorizedError`) from services and repositories — never raise
  `HTTPException` outside of `router.py`.
- A single exception handler in `main.py` maps custom exceptions to HTTP responses.
  This keeps services testable without FastAPI in the loop.
- Never use a bare `except:`. Catch specific exceptions; log unexpected ones with
  context before re-raising.

## 5. Logging

- Use the standard `logging` module configured in `core/logging.py`, not `print()`.
- Log at service-layer boundaries: entry of significant operations (e.g. "starting
  document processing for doc_id=...") and on caught exceptions.
- Never log secrets, passwords, or full JWT tokens.

## 6. Database & Migrations

- Every schema change goes through an Alembic migration — no manual `ALTER TABLE`,
  no `Base.metadata.create_all()` in application code outside of test fixtures.
- Migration filenames are autogenerated by Alembic; never hand-edit a migration's
  revision ID.
- Foreign keys are mandatory wherever a relationship exists — no implicit joins via
  matching column names without an FK constraint.

## 7. API Design

- All endpoints are versioned under `/api/v1/...`.
- Pydantic schemas for request/response are separate from ORM models — never return an
  ORM model directly from a route.
- List endpoints are paginated by default (use `shared/pagination.py`); no endpoint
  returns an unbounded list.
- Use proper HTTP status codes (`201` for creation, `204` for delete, `404` vs `422`
  distinction, etc.) — don't default everything to `200`.

## 8. Testing

- Every `service.py` requires unit tests with the repository mocked.
- Every `repository.py` requires integration tests against a real (test) database.
- Every `router.py` requires API tests using FastAPI's `TestClient` / `httpx.AsyncClient`.
- A module is not "done" (per the Development Process in the dev instructions) until
  all three test layers exist for it.

## 9. Security Checklist (apply per feature, not just globally)

- Passwords hashed with `bcrypt` (via `passlib`), never stored or logged in plaintext.
- All file uploads validated for type and size before processing; stored outside the
  web root with a generated filename (never trust the client-provided filename).
- All `user_id`-scoped queries filter by the authenticated user — verify this
  explicitly in tests (a common bug class is forgetting the `WHERE user_id = ?` filter).
- Secrets only ever come from environment variables (`config/settings.py` via
  `pydantic-settings`), never hardcoded, never committed — `.env` is gitignored,
  `.env.example` documents required keys with placeholder values.

## 10. Definition of Done (per feature)

A feature is complete only when all of the following are true:

1. Database migration written and applied.
2. Repository, service, router implemented per the layering rules above.
3. Pydantic schemas validate input/output.
4. Unit + integration + API tests pass.
5. Docstrings present on public functions.
6. Module documented (purpose, key design decisions) if non-trivial.
7. No `print()`, no bare `except`, no hardcoded secrets, no TODO left unexplained.

This mirrors the "Code Reviews" checklist in the project's development instructions —
treat that section as the final gate before marking a feature complete.
